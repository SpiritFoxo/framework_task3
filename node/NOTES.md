# Пояснение по критичным настройкам

## Какие настройки считаются критичными и почему

### 1. Режим работы (`mode`)

**Критичность: высокая.**

Режим определяет, насколько подробно служба раскрывает внутренние детали при ошибках. В учебном режиме клиент получает точное сообщение исключения — это удобно при разработке, но в боевой среде такое поведение помогает атакующему: он узнаёт, какие поля не прошли валидацию, какова внутренняя структура данных, и может итеративно подбирать запросы. Боевой режим возвращает нейтральное `"Ошибка обработки запроса"` во всех случаях.

Переключение реализовано исключительно через настройку — в `server.js` нет условных блоков с хардкодом для окружений, только чтение значения `mode` из конфига.

### 2. Список доверенных источников (`trustedOrigins`)

**Критичность: высокая.**

Пустой список или отсутствие проверки означает, что браузер с любого сайта может отправлять запросы к API от имени пользователя (CSRF). Именно поэтому:

- При пустом списке валидация **блокирует запуск** — запускать службу «без ограничений» нельзя намеренно.
- Каждый адрес проверяется как корректный URL с протоколом `http:` или `https:`. Опечатка вроде `htps://mysite.ru` отловится на старте, а не появится дырой в prod.
- Заголовок `Access-Control-Allow-Origin` выставляется только для известных источников; неизвестные запросы не получают разрешения.

### 3. Порт (`port`)

**Критичность: средняя.**

Невалидный порт (0, отрицательный, >65535, не целое) приводит к тому, что `server.listen()` либо кидает исключение, либо слушает неожиданный адрес. Ранняя проверка в `validateConfig` позволяет увидеть проблему до попытки запуска, с понятным сообщением, а не трассировкой стека Node.js.

### 4. Лимиты частоты запросов (`rateLimits`)

**Критичность: средняя.**

Нулевой или отрицательный лимит формально означает, что все запросы блокируются сразу. Поэтому:

- `readPerMinute` и `writePerMinute` должны быть положительными целыми.
- Лимит записи не должен превышать лимит чтения — иначе конфигурация противоречит сама себе (создавать элементы можно чаще, чем читать список).

Отдельные лимиты для чтения и записи позволяют гибко защитить дорогостоящие операции: создание элемента обычно требует больше ресурсов, чем простое чтение, поэтому его лимит ниже.

## Как обеспечено раннее обнаружение ошибок конфигурации

Вся проверка сосредоточена в функции `validateConfig`, которая вызывается **до** `server.listen()`. Если возвращается хотя бы одна ошибка, процесс завершается с кодом 1 и выводит список всех найденных проблем сразу (не только первую). Это позволяет исправить все ошибки за один цикл, а не запускать сервер несколько раз.

Такой подход снижает риски двумя способами:

1. **Операционный риск**: некорректно настроенная служба никогда не запустится в prod, вместо того чтобы молча работать с небезопасными параметрами.
2. **Риск эксплуатации**: ошибки, связанные с безопасностью (пустой список источников, некорректные URL), обнаруживаются на этапе деплоя, а не во время первого запроса от атакующего.

## Защитные заголовки ответа

Добавлены четыре заголовка (минимум два по заданию):

| Заголовок                  | Защита от                                                        |
|----------------------------|------------------------------------------------------------------|
| `X-Content-Type-Options: nosniff`  | MIME-sniffing: браузер не будет угадывать тип контента  |
| `X-Frame-Options: DENY`            | Clickjacking: страницу нельзя встроить во фрейм         |
| `Cache-Control: no-store`          | Кэширование чувствительных ответов промежуточными узлами |
| `Pragma: no-cache`                 | То же для HTTP/1.0 прокси                               |

Заголовки применяются ко всем ответам через единую функцию `applySecurityHeaders`, которая вызывается в начале каждого запроса — до маршрутизации.
